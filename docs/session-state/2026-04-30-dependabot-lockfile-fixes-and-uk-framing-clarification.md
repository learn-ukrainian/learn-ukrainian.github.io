### Arc 2 — Pre-existing pip-check issues → cascade fix on lockfile

After dependabot merges, surfaced two `pip check` warnings:
1. `marker-pdf 1.10.2 has requirement anthropic<0.47.0,>=0.46.0, but you have anthropic 0.89.0` (pre-existing — anthropic 0.89 already violated marker-pdf's <0.47 cap)
2. `torch 2.10.0 is not supported on this platform`

User asked Claude to fix them.

**Investigation revealed both are surface symptoms of a deeper problem:** `requirements-lock.txt` is generated by `pip freeze` (per the file's header comment), which captures whatever's in the venv but does NOT validate cross-package constraints. Each dependabot bump touches one package and doesn't re-resolve the full graph. Result: incompatibilities accumulated silently.

Trying to refresh the venv from the lockfile failed in 4 different places, exposing real breakage (not just pip-check noise):

- `huggingface_hub 1.8.0` ≠ `transformers 4.57.6` (`<1.0`) → **was breaking sentence_transformers + transformers imports at runtime** when the venv eventually rebuilt
- `mando 0.8.2` ≠ `radon 6.0.1` (`<0.8`)
- `datasets 4.6.0` ≠ `multiprocess 0.70.19`
- `dill 0.4.0` couldn't satisfy datasets+multiprocess intersection
- Multiple transitives missing entirely: joserfc, pypdf, typer, curl_cffi, docstring_parser, yt-dlp

CI's `pip install --no-deps -r requirements-lock.txt` masked these — `--no-deps` skips the resolver, so it installed all listed pins without checking they actually work together. The bug only surfaces on a clean venv build OR a strict-resolver install.

**Fixed (commit `143c955c40`):**
- Updated lockfile pins to a mutually-consistent set
- Added the previously-missing transitive deps
- Removed accidentally-installed `tokenize_uk` pip package (project policy: vendored at `scripts/linguistics/tokenize_uk.py` with local ABBRS extensions — explicitly forbidden as a pip dep per `requirements.txt:21-22`)
- Added inline section to `requirements.txt` documenting the 3 remaining `pip check` warnings as known-harmless: marker-pdf vs anthropic + regex (we don't use `use_llm=True` mode); pycookiecheat vs cryptography (Chrome cookie API runtime-verified at cryptography 47)

**Verified post-fix:**
- `pip install --no-deps --dry-run -r requirements-lock.txt` clean
- 6993 tests collect (was 6992 + 1 collection error from huggingface_hub mismatch)
- Critical imports OK: torch 2.11.0 + MPS, sentence_transformers, transformers, marker-pdf, pycookiecheat
- Final `pip check` is just the 3 documented harmless warnings

**Filed #1634** for the underlying problem. Today's commit is a bandaid — the proper fix is migrating off `pip freeze` to a real resolver (pip-tools / uv / poetry). Recommended: pip-tools as default. Until #1634 ships, every dependabot pip PR risks adding a new inconsistency.

## Worth-knowing details

### Verbal slippage discipline (Framing A vs B)

Lesson from Arc 3: the same architectural decision (wikis as Ukrainian source content for retrieval) admits two framings that LOOK identical but undersell vs overdeliver on the wiki rebuild's value. Future Claude sessions:
- If you say "the wiki is just retrieval plumbing" — STOP. That's Framing B.
- If you say "the wiki rebuild IS the UK content-creation track, generating decolonized yabluko-equivalent pedagogical artifacts" — that's Framing A.
- Same architecture. Different quality bar. Framing B drops citation audits, register reviews, standalone-publishability scrutiny. Framing A keeps them as gates.

The `memory/l1-uk-corpus-bootstrap.md` "Framing trap" section is the canonical reference now.

### Lockfile-as-pip-freeze is structurally fragile

Today's Arc 2 was the third or fourth time recently that lockfile coherence has bitten us (the others were less visible — packages stale-but-installable hid them). The pattern:

1. Dependabot bumps one package + its lockfile entry
2. Local resolver doesn't validate cross-deps
3. CI's `--no-deps` install hides the staleness
4. A clean venv rebuild OR a strict-resolver install eventually surfaces it

Until #1634 ships, **after every dependabot pip PR merges, run `pip install --no-deps --dry-run -r requirements-lock.txt`** to detect new inconsistencies before they hide. Add this as a one-line CI step? Possibly worth a small PR — would catch new conflicts as they're introduced rather than weeks later.
