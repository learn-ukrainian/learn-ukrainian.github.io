Secret-leak incident 2026-05-10 morning

What happened

While diagnosing whether GEMINI_API_KEY was properly commented in ~/.bash_secrets, I ran grep -nH "GEMINI_API_KEY" "$full" across shell startup files and printed the matched line verbatim. .bash_secrets:3 contained #export GEMINI_API_KEY="AIza..." — the value got transcript-leaked.

The same diagnostic command had sed 's/=.*/=<REDACTED>/' applied to env output but NOT to file-grep output. Inconsistent sanitization within a single command IS the failure mode.

User had to rotate.

Prevention shipped

  • New autopsy: docs/bug-autopsies/secret-leakage.md — decision tree for "is var X set" without leaking value (use env -i bash -c 'source ...; [ -n "${VAR:-}" ]'; never grep credential files for set-ness checks).
  • New MEMORY rule #M-5 (above #M-4 in priority): "Never print matched lines from credential files; always pipe through cut -d= -f1 or sed 's/=.*/=<REDACTED>/'; inconsistent sanitization within a single diagnostic IS the failure mode."
  • Sibling search: grepped committed scripts for the same anti-pattern — none found in source. The leak path is exclusively agent ad-hoc commands. Fix is behavioral (rule + autopsy), not code.
