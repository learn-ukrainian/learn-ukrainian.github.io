The codex-tools fix — root cause + sanitizer + smoke test result

Root cause — the v7 dispatch path emits this to codex exec:

-c mcp_servers.sources.type="streamable-http"
-c mcp_servers.sources.url="http://127.0.0.1:8766/mcp"

Codex CLI's mcp_servers.<name>.* schema does not have a type field. The user's working ~/.codex/config.toml has only url. Unknown field silently breaks server registration.

Direct evidence from ~/.codex/sessions/2026/05/08/rollout-2026-05-08T11-54-40-*.jsonl line 10: “requested mcp__sources__... tools 'are not exposed in this session'”. The model itself articulated the missing catalog.

The fix in scripts/agent_runtime/tool_config.py: a sanitizer keeping only the codex-recognized fields.

_CODEX_MCP_SERVER_FIELDS = frozenset({
    "url", "command", "args", "env", "bearer_token_env_var",
})

def _codex_sanitize_server_config(server_config: dict) -> dict:
    return {k: v for k, v in server_config.items() if k in _CODEX_MCP_SERVER_FIELDS}

Wired into _codex_mcp_servers for both the explicit-request and no-request branches.

Positive runtime gate in scripts/build/linear_pipeline.py: when a *-tools writer ends with tool_calls_total == 0, raise LinearPipelineError("MCP_TOOLS_NEVER_INVOKED", ...). Pre-flight mcp_config_resolved.resolution_status='ok' only verifies config string resolution, not runtime tool injection. The positive gate is the actual safety net.

Smoke test in worktree (Codex's own verification before push): codex exec ... "call mcp__sources__verify_words for word 'кіт'" → returned “кіт → lemma кіт, noun”. Real call, real result.

Full converged report: audit/codex-tools-review-2026-05-08/REPORT.md (still markdown — to be backfilled to HTML).
