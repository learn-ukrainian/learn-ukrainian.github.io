### Loader API (what Step 5 consumes)

```python
from scripts.build.universal_rules_registry import load_applicable_rules, Rule

rules: list[Rule] = load_applicable_rules(
    level="a1",
    track="core",
    activity_profile="default",
    slot="shared.contract",  # or "writer.body" / "writer.preamble" / "reviewer.rubric" / None

# returns topologically sorted: every `depends_on` precedes its dependents
# tie-break: filename-alphabetical
```

Validated invariants (enforced by `_parse_fragment` + tests):
- Filename stem MUST equal frontmatter `id` (so `R-VESUM-ALL-WORDS.md` must have `id: R-VESUM-ALL-WORDS`).
- `id` must start with `R-` (NO `#` prefix in frontmatter; the `#` is reconstructed via `Rule.telemetry_id`).
- `slot` ∈ {`writer.preamble`, `writer.body`, `reviewer.rubric`, `shared.contract`}.
- `applies_to.levels` ⊆ {a1, a2, b1, b2, c1, c2, all}; `tracks` ⊆ {core, seminar, all}; `activity_profiles` is free-form.
- `depends_on` entries validated against the FULL registry, NOT the filtered subset → a filtered-out parent doesn't block the dependent, but a missing-from-registry parent always raises `MissingDependencyError`.

### Q2 — Schema additions before Step 5 commits to it

PR #2394 explicitly invited reviewers to add fields now (cheap to change before generator references the schema):

**Default decision next session:** schema stays as-is (id, description, applies_to, slot, depends_on). Step 5 commits.

If user disagrees, raise BEFORE Step 5 fires — much cheaper than a follow-up PR after generator code references the schema.

### Q3 — 3-way slot cut for A1 writer.body rules — keep or promote to shared.contract?

The 3 A1-only `writer.body` rules (R-AUDIENCE-LANGUAGE-A1, R-SINGLE-VOICE-A1, R-GRAMMAR-TERMS-A1) are NOT emitted to the reviewer prompt under the current scheme. They're "writer instructions about output style" — the writer is told to "stay in English at A1", "use single teacher voice", "use real grammar terms".

**Cursor's single-source-emission principle** (from the multi-agent ADR discussion): *"three hand-maintained descriptions of the same contract drift apart."* The reviewer arguably should know the writer was told these — otherwise it can't catch when the writer fails to follow them.

**Options:**
- (a) Keep 3-way cut: writer.body stays writer-only. Reviewer doesn't see these rules. (Current shipped state.)
- (b) Promote all 3 to `shared.contract`: reviewer sees the writer's instructions and can score against them.
- (c) Add `reviewer.rubric` rules that mirror them with reviewer-specific phrasing (would have wanted Q2's `reviewer_phrasing` field).

**Orchestrator recommendation:** (b) — promote to `shared.contract`. Reviewer rubric is dimensional; the current `tone` reviewer at A1 was over-penalizing UK connectors in EN prose because it didn't know the charter said "EN explanation, UK as target only." Giving the reviewer the same rule the writer saw closes that gap.

**Default decision next session:** promote the 3 to `shared.contract` if user agrees. Touches 3 fragment files + maybe a comment in the generator doc. Trivial PR before Step 5 fires.

## Behavioral lessons from this session

1. **Parallel codex dispatches are GO when paths don't overlap.** Steps 1+2+3 were designed in Pt 14 to be independent — fired all 3 in parallel, no cross-contention, all 3 PRs opened within 6 minutes of each other. The `--silence-timeout 5400` from Pt 14's lesson worked perfectly: zero kills, all 3 finished cleanly. Codex post-PR-open idle pattern (#M-8) reproduced: dispatch shows `running` for ~45 min after task_complete event because codex CLI hangs around. **`gh pr list` is the source of truth, not `/api/delegate/active`.**

2. **Monitor filter discipline matters.** First Monitor filter (broad regex matching `git push`, `gh pr create`, etc. anywhere in the JSONL line) generated event spam from tool_call payloads containing those literal strings. **Tightened to `"payload":\{"type":"task_complete"|"shutdown"|"error"`** — exactly 4 events fired across the whole session (one per dispatch terminal state). Per the Monitor docstring: *"the filter should be selective — but selective means 'the lines you'd act on,' not 'only good news.'"* Right pattern is structural JSON-shape match, not keyword.

3. **`review / review` cancellation is the Gemini auto-review timeout pattern.** All 4 PRs hit the exact same 7m11s `cancelled` conclusion. Per #M-0.5 / Gemini-Dispatch advisory layer: cancelled = no findings delivered = not blocking. Documented elsewhere but worth re-noting: do NOT treat `review / review = cancel` as a substantive failure.

4. **Claude headless completes design tasks faster than the brief estimates.** Step 4 brief estimated 1-2 days; actual active dispatch time was ~10 minutes for 15 fragments + 289-LOC loader + 384-LOC tests + 149-LOC design doc + answers to 4 design questions. Either the brief over-estimated, or Opus 4.7 was operating at full velocity. Useful calibration for sizing future claude-headless dispatches.

5. **Worktree-removal-after-merge requires explicit cleanup.** `gh pr merge --squash --delete-branch` deletes the REMOTE branch but fails to delete the LOCAL branch when a worktree still references it. Pattern: after merge, `git worktree remove --force <path> && git branch -D <branch>`. This recurred 4 times this session — solid reason to standardize a post-merge cleanup helper.

6. **Stale `.git/index.lock` files from background `git fetch` calls.** Observed twice this session — background CI watchers triggered concurrent fetches that left zero-byte lock files. Clean rm + retry resolves. If this pattern persists, consider centralizing fetch through a serialized helper or removing the lock-file pattern from concurrent watchers.
