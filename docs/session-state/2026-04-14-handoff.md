### PRIORITY 0 — Landing sites real-time module status (CRITICAL)
**User explicit priority: "landing sites of curriculums give real time status of modules, and not stale data"**

The curriculum landing pages (starlight/) and dashboards must reflect CURRENT module status, not cached/stale data. Investigation needed:

**Check these paths:**
- `starlight/src/content/docs/` — static module pages, built from audit status at build time
- `scripts/build/build_landing_pages.py` — landing page generator
- `scripts/api/module_dashboard.py` — module dashboard API (flagged as bypassing `read_status()` in Codex review)
- `scripts/api/dashboard_router.py:181-182` — also bypasses `read_status()` centralizer

**Confirmed issues (5 minute scan):**
1. Every `.mdx` file has a `build_status: draft|validated|reviewed` frontmatter field
2. `build_status` is written by `scripts/generate_mdx/core.py:94-111` at **publish time only** (from phase completion, NOT from current audit status)
3. If audit fails AFTER publish, `.mdx` still shows `validated`/`reviewed` (stale)
4. If audit improves (module now passes) without a re-publish, `.mdx` shows old status (stale)
5. Dashboard API endpoints (`module_dashboard.py`, `dashboard_router.py:181-182`) bypass `read_status()` so freshness detection is lost
6. No client-side refresh — user must reload the page to see any change

**Solution options (pick one with user):**
- **Option A (static, force rebuild):** Add a file-watcher hook that re-publishes `.mdx` whenever `status/{slug}.json` changes. Simple but costly on every status write.
- **Option B (client-side live):** Landing pages fetch current status from Monitor API (`/api/state/module/{track}/{num}`) on page load. Keep `build_status` in frontmatter as initial/fallback. Add a small status badge component that pulls live.
- **Option C (SSE stream):** Monitor API exposes `/api/state/module/{track}/{slug}/stream` as Server-Sent Events. Landing page subscribes and updates without reload.

**Recommended: Option B.** Simplest, no extra infra, works offline (falls back to frontmatter), doesn't require rebuild on every status tick. Add a `<LiveStatus slug="weather" level="a1" />` component that fetches from the API and updates a badge in real time.

**Test strategy (user explicit):** Implement on **A1 first**. A1 has 54/55 modules passing so the component can be verified against known-good data. Once working on A1, extend to A2/B1/B2/C1/C2.

**Approach:**
1. First: diagnose current data flow — how does a status file change propagate to landing page?
2. Identify stale data sources (static generation? cached API? no invalidation?)
3. Fix via: (a) force landing page rebuild on status change via hook, or (b) client-side status fetch from API, or (c) SSE stream from Monitor API
4. Ensure all status reads go through `read_status()` so freshness detection works

**This must be fixed before A2 rebuild** — otherwise users will see outdated failure statuses on freshly-rebuilt passing modules.

## Cross-agent review pattern (proven in previous session)

The adversarial cycle finds bugs neither agent catches alone:
1. **Codex reviews code** → finds bugs → **Codex fixes** → **Gemini reviews fixes**
2. **Gemini reviews code** → finds bugs → **Gemini fixes** → **Codex reviews fixes**

Previous session: Codex approved Gemini's fixes on the 5th iteration. Each rejection found a real deeper bug. Don't shortcut the cycle — Codex kept finding real edge cases.

### Template for fix task
```
FIX TASK: [summary]
You found these issues:
1. [file:line] — [description] — [suggested fix]
2. ...

Edit files, run ruff, commit with #1228. End with [DONE].
```

### Template for review task
```
REVIEW [agent]'s fixes (commit [SHA]). He fixed [N] issues:
1. [summary of fix]
...
Check: correct? Edge cases? APPROVE or REJECT.
```
