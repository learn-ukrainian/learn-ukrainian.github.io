Key synthesis points (full doc at docs/best-practices/harness-engineering.md):

  • Definition: Harness engineering = designing the environment (tools, abstractions, in-repo knowledge, mechanical guardrails) so every agent turn succeeds. Sits above prompt+context engineering at the codebase layer.
  • OpenAI's numbers: ~1M LOC in 5 months, ~1,500 PRs, 3 engineers → 7, 3.5 PRs/engineer/day, single Codex runs working 6+ hours on one task.
  • Symphony's pivot: stop tracking work via coding sessions; track via project-management state. Each open Linear ticket → dedicated agent workspace; ticket terminal → workspace cleaned. 500% increase in landed PRs reported.
  • Key Symphony quote: "Treating agents as rigid nodes in a state machine doesn't work well. Models get smarter... So we eventually moved toward giving agents objectives instead of strict transitions." — independent convergence on the /goal pattern we shipped 2026-05-12 morning.
  • Critical caveat for us: their system is built for software where bad output is cheap to revert (re-run, re-PR, rollback). Module content is the opposite regime — bad pedagogy trains bad habits into real learners. The harness principles map cleanly to our scripts/CI/infrastructure but do NOT port to module content production.

Mapping table (full version in the harness-engineering.md doc) shows we already do most of the harness work by trial and error — CLAUDE.md as ToC, structured docs/ system of record, mechanical lints, worktree-isolated dispatches, bug autopsies, two-tier handoffs, /goal rule. The OpenAI posts gave it a name and a reference architecture.
