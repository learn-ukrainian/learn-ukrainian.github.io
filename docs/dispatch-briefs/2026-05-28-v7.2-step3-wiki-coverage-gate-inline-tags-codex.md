## Goal

Update `scripts/audit/wiki_coverage_gate.py::parse_implementation_map()` to accept both the legacy nested form AND the V7.1 self-closing inline form of `<implementation_map>` tags.

## Background

Today's V7.1+codex build at `.worktrees/builds/a1-my-morning-20260528-122552/curriculum/l2-uk-en/a1/my-morning/module.md` produced 1505w of clean A1 lesson content with all 18 obligations tracked in implementation_map.json — but **wiki_coverage_gate returned 0/18 (coverage_pct: 0.0, hard_fail: true)** with reason `implementation_map_missing` on every obligation.

Root cause: `parse_implementation_map()` looks for nested-block shape:
```
<implementation_map>
  <row obligation_id="step-1" artifact="..." location="..." treatment="..." />
  <row obligation_id="step-2" .../>
</implementation_map>
```

But the V7.1 writer emits self-closing inline tags directly:
```
<implementation_map obligation_id="step-1" artifact="module.md" location="Дієслова на -ся" treatment="sequence_step" />
<implementation_map obligation_id="step-2" artifact="module.md" location="..." treatment="sequence_step" />
```

Both shapes carry identical information. The gate should accept both during the V7.2 transition (and ideally choose ONE canonical form at the end of Step 6's single-source emission work).

## Required change

In `scripts/audit/wiki_coverage_gate.py::parse_implementation_map()`:

1. Add a second regex pass that matches self-closing inline `<implementation_map obligation_id="..." artifact="..." location="..." treatment="..." />` tags.
2. For each matched inline tag, extract the four attributes and add to the result map under `obligation_id`.
3. Merge with results from the existing nested-block parse (so both shapes work in the same document).
4. Document in a comment that both shapes are accepted; the V7.2 prompt generator (Step 6) will standardize on one canonical shape.
