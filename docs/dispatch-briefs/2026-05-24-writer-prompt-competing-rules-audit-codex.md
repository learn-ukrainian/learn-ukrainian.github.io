## Why this audit

Two m20 a1/my-morning build attempts (#5 and #6, 2026-05-23) both failed because the writer (claude-tools) cited Захарійчук **Grade 4 pp.162/163** in `resources.yaml` when `plan.references` only listed **Grade 1 pp.24/52**. The writer's behavior was:

1. Ignored the new `#R-TEXTBOOK-30W` rule Step A ("use the chunk_id from plan.notes directly").
2. Trusted the upstream Knowledge Packet's `S1 anchor` (which pointed to Grade 4) over the plan.
3. Fabricated additional references not in plan.

The orchestrator made TWO prompt rewrites adding HARD STOP wording to `#R-TEXTBOOK-30W` — neither moved the writer's behavior. This suggests **a competing rule elsewhere in the writer prompt is fighting `#R-TEXTBOOK-30W`** (e.g., a rule encouraging "enrich the lesson with additional references" or treating Knowledge Packet anchors as authoritative citations).

The handoff names this as **Option B**: audit the writer prompt for competing rules.

Companion fix Option C (deterministic `plan_reference_match` gate) is shipping separately on PR #2257. Both fixes are additive: the gate catches fabrication post-emit; this audit identifies why the writer fabricates in the first place.

## What to deliver

A focused report at `audit/2026-05-24-writer-prompt-competing-rules.md` answering 4 questions:

### Q1: Where does `#R-TEXTBOOK-30W` live in the rendered prompt vs. competing rules?

Read `scripts/build/phases/linear-write.md`. Find the `#R-TEXTBOOK-30W` block (Steps A-D introduced 2026-05-23). Then grep for sections that mention any of:

- "enrich", "extend", "augment", "additional reference", "more sources"
- "Knowledge Packet", "packet anchor", "S1", "S2", "S3" (anchor labels)
- "Cite", "citation", "reference" (every occurrence — context matters)
- "resources.yaml" emit rules (what goes in it)
- "implementation_map" obligations (Knowledge Packet → emit contract)

For each section, quote the relevant 3-10 lines and rate whether they CONTRADICT, REINFORCE, or are NEUTRAL toward "use plan.notes chunk_id, do not invent references."

### Q2: Does the Knowledge Packet template push the writer toward Grade-4 anchors?

The Knowledge Packet is generated by `scripts/build/seed_implementation_map.py` (or similar) and rendered into the writer prompt. Find the rendered Knowledge Packet block for a1/my-morning at `.worktrees/builds/a1-my-morning-20260523-220712/curriculum/l2-uk-en/a1/my-morning/knowledge_packet.md` (preserved per #M-10). Check:

- Does the Knowledge Packet include Grade 4 references that are NOT in `plan.references`?
- If yes, does the writer prompt explicitly say "Knowledge Packet anchors are reference material only, NOT citation candidates"?
- If no such guard exists, that's a finding.

### Q3: Does the schema for `resources.yaml` allow citations outside `plan_references`?

Read `scripts/build/linear_pipeline.py` — find the `WRITER_JSON_SCHEMAS["resources.yaml"]` block. Does the schema enforce "every textbook-role entry must have a chunk_id from plan.references"? Or is it permissive?

(Note: the `citations_resolve` gate post-emit enforces this for the non-`packet_chunk_id` path. But the writer doesn't see that gate until after emit. Schema is the only PRE-EMIT enforcement surface.)

### Q4: What's the smallest prompt change that would fix this without touching the gate?

Based on Q1-Q3 findings, propose 1-3 concrete diff hunks (file:line + old/new text) that would meaningfully redirect the writer's behavior. Prioritize:

- Moving the strongest "use plan only" guidance EARLIER in the prompt (LLMs weight early tokens more).
- Negating the strongest competing rule (e.g., if there's a "enrich" instruction, qualify it with "ONLY from plan.references").
- Adding a verbatim example: "Plan.references: [Grade 1 p.24]. resources.yaml MUST cite ONLY chunk_id from that entry. Do NOT add Grade 4 references even if the Knowledge Packet anchor S1 points to one."

Do NOT actually apply the changes — this is a read-only audit. Just propose them in the report.
