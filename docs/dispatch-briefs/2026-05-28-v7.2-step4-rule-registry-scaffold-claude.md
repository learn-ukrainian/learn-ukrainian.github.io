## Goal

Design + scaffold the universal rules registry at `scripts/build/universal_rules/`. Extract the cleaned (post-Step-1) static content from `scripts/build/phases/linear-write.md` into ~20-30 fragment files keyed by `#R-*` IDs, each with YAML frontmatter specifying which levels/tracks/activity-profiles the rule applies to. Build a registry loader. No generator wiring yet — that's Step 5.

## Required surface

1. **Design the fragment file schema.** Each fragment lives at `scripts/build/universal_rules/<RULE_ID>.md` with frontmatter:
   ```yaml
   ---
   id: R-VESUM-ALL-WORDS
   applies_to:
     levels: [a1, a2, b1, b2, c1, c2, all]
     tracks: [core, seminar]
     activity_profiles: [all]
   slot: "writer.body" | "writer.preamble" | "reviewer.rubric" | "shared.contract"
   depends_on: [R-OTHER-ID, ...]
   ---
   ```
   The `slot` field indicates whether the fragment goes into the writer prompt, the reviewer prompt, or both (`shared.contract`). Some rules need to appear in both (writer is asked to do X; reviewer checks X) — those live in `shared.contract` and are emitted to both prompts by the generator.

2. **Extract fragments from `linear-write.md`.** After Step 1's cleanup landed, identify the `#R-*` rules in the existing static template. For each one, create a fragment file at `scripts/build/universal_rules/<R-NAME>.md`. Preserve rule semantics exactly — this is mechanical extraction with metadata addition, not rule rewriting.

3. **Build the registry loader.** Create `scripts/build/universal_rules/__init__.py` (or `scripts/build/universal_rules_registry.py`) that:
   - Discovers all fragment files in `scripts/build/universal_rules/`
   - Parses frontmatter
   - Exposes `load_applicable_rules(level: str, track: str, activity_profile: str, slot: str) -> list[Rule]`
   - Returns rules sorted by dependency order (topological sort over `depends_on`)
   - Returns plain markdown body for each rule, ready to concatenate into prompt

4. **Do NOT wire the generator yet.** That's Step 5. This Step's deliverable is just the registry + loader + extracted fragments. The existing `linear-write.md` template remains the active source for the writer prompt for now.

5. **Comprehensive unit tests** for the registry loader: fragment discovery, frontmatter parsing, level/track filtering, topological dependency ordering, edge cases (missing dependency, circular dependency).

## Design questions to answer in the PR body

1. **Where do `#R-AUDIENCE-LANGUAGE-A1` and similar level-specific rules live?** They have `applies_to.levels: [a1, a2]`. The frontmatter handles this. Confirm via the test suite.

2. **What about rules that mirror in writer + reviewer (e.g., `#R-NO-CHILDREN-PRIMARY-QUOTES` — writer is told not to use them, reviewer is told to flag them)?** Use `slot: shared.contract`. The generator emits to both prompts. Verify with a test.

3. **What about composition slots that need lesson data inlined (e.g., decolonization contrast pair from the wiki)?** Those are not universal rules but composition directives. They live OUTSIDE the registry (in the generator's composition logic, Step 5). Document this boundary explicitly in the PR.

4. **How to handle rule ordering across fragments?** Topological by `depends_on`, fallback to filename alphabetical. Document this in the loader's docstring.
