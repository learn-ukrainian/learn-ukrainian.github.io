## What to build

A linter that detects "menu" antipatterns in markdown text — specifically the patterns that violate `MEMORY.md` rule **#0I — DON'T STACK MICRO-DILEMMAS, DECIDE FOR THE USER**.

**Forbidden patterns**:
- "(a) vs (b)" / "(a) ... or (b)" presented as a sign-off menu
- Numbered "1) Do X 2) Do Y 3) Do Z — which?" with a question at the end
- "Want me to A, B, or C?" / "Should I do X or Y?"
- "Sign off on these N options:" followed by a numbered list

**Allowed (must NOT trigger)**:
- Acceptance criteria lists
- Step-by-step plans
- Status reports listing what was done
- ADR option enumerations (where the doc itself is the decision artifact)

## Acceptance criteria (numbered, all required)

1. **New script** `scripts/audit/lint_anti_menu.py` with:
   - `argparse` CLI: `--text PATH` (required, file to scan) OR `--stdin` (read from stdin)
   - Regex-based detection. **Heuristic, not perfect** — false positive rate < 5% on a corpus of 20 real handoff docs from `docs/session-state/`.
   - Allowlist mechanism: lines following an `## Acceptance criteria` heading, lines inside fenced code blocks, lines in tables, and lines preceded by `Done:` / `Status:` / `Plan:` are exempt.
   - Output: per-violation report `path:line: anti-menu pattern detected — '<matched-snippet>'`
   - Exit 0 if clean, 1 if any violations
   - Conforms to `cli-help-standard.md`
2. **Tests** in `tests/audit/test_lint_anti_menu.py`:
   - 3 fixture texts that SHOULD trigger (clear menu antipatterns)
   - 3 fixture texts that should NOT trigger (acceptance criteria, ADR options inside code fence, status lists)
   - 1 fixture: a real-shaped handoff doc (~50 lines) that should NOT trigger
   - Assert correct exit codes + that the matched-snippet output is informative
3. **Document the regex shapes** in module-level docstring so future maintainers can extend.
4. `ruff check` clean.
5. `.venv/bin/pytest tests/audit/test_lint_anti_menu.py -x` passes.
6. `docs/SCRIPTS.md` entry.
