# Plan: Two-Phase Activities + Self-Improving Prompt Loop

## Context

The current pipeline generates prose and activities in one pass, then runs expensive Claude fix loops when quality is low. Problems:
1. Fix loops rarely converge — they patch symptoms, not root causes
2. Activities depend on final prose, but are generated alongside it
3. Humans can't review prose before activities are built (wasted effort if prose needs changes)
4. When prompts produce bad output, we fix the output instead of fixing the prompt

This plan introduces: (a) two-phase activity workflow with human review between phases, (b) textbook-sourced activities via RAG, (c) a self-improving loop where Claude + Gemini fix the TEMPLATE instead of the content.

---

## Phase Sequence Change

**Current:** `research → discover → content → activities → validate → [review] → mdx`

**New:** `research → discover → content → validate → [review] → activities → mdx`

Activities move AFTER review. The content phase outputs activity PLANS (type + description) alongside prose. MDX renders placeholders until activities are built. This is simpler than adding new phase names — we just reorder and make activities aware of plans.

**Why not new phase names?** Adding `activity_plan`, `validate_prose`, `validate_full` creates migration headaches for existing state.json files and complicates `--restart-from`. Instead:
- `content` generates prose + activity plans (new `activity-plans.yaml` file)
- `validate` and `review` run on prose only (activities not yet built)
- `activities` runs AFTER review, using plans + RAG + final prose
- MDX handles both states: plans-only → placeholders, built → components

### CLI Usage

```bash
# Phase 1: Build prose + activity plans, validate, review
.venv/bin/python scripts/build_module_v5.py a1 1 --stop-before activities

# Human reviews prose in MDX (with activity placeholders)
# Human approves or requests changes

# Phase 2: Build activities from plans + RAG
.venv/bin/python scripts/build_module_v5.py a1 1 --restart-from activities
```

## 1. Activity Plans Format

**File:** `curriculum/l2-uk-en/{track}/activities/{slug}-plans.yaml`

```yaml
- type: fill-in
  description: "Complete words by blending taught letters into syllables"
  item_count: 8
  focus: "М+А→МА, complete partial words"
  textbook_source: "Adapted from Bolshakova Grade 1, syllable blending"

- type: match-up
  description: "Match Ukrainian letter to its English sound equivalent"
  item_count: 10
  focus: "False friends: Н≠H, С≠C"
```

Generated by Gemini during content phase via `===ACTIVITY_PLANS_START===` / `===ACTIVITY_PLANS_END===` delimiters.

## 2. MDX Placeholder Rendering

**New component:** `starlight/src/components/ActivityPlaceholder.tsx`

Simple card showing type badge, description, and item count. Non-interactive. Styled to look like a planned-but-not-built activity.

**MDX generator change** (`scripts/generate_mdx/core.py`, line ~270):

```python
# Current:
if yaml_activities:
    activities_content = yaml_activities_to_jsx(yaml_activities, ...)

# New:
if yaml_activities:
    activities_content = yaml_activities_to_jsx(yaml_activities, ...)
elif activity_plans:
    activities_content = activity_plans_to_jsx(activity_plans)
```

`activity_plans_to_jsx()` renders each plan as:
```jsx
<ActivityPlaceholder type="fill-in" description="..." itemCount={8} focus="..." />
```

## 3. Activity Build with Textbook RAG

When `phase_activities` runs (Phase 2), it:

1. Loads `{slug}-plans.yaml`
2. For each plan, searches RAG for matching textbook exercises:
   ```python
   mcp__rag__search_text("вправа класифікація голосних приголосних", limit=3)
   ```
3. Passes plans + RAG results as context to new template `activity-build.md`
4. Gemini ADAPTS real exercises to our schema types, not invents from scratch
5. Extracts via existing `===ACTIVITIES_START===` / `===ACTIVITIES_END===` delimiters

**Template:** `claude_extensions/phases/gemini/activity-build.md`

Key instruction: "You are given activity plans AND real textbook exercises from Ukrainian school books. Adapt the textbook exercises to match the plan's type, focus, and item count. Do NOT invent exercises — transform the real ones."

## 4. Self-Improving Prompt Consultation

**When:** After Claude reviews and finds issues that are template-caused (not one-off content errors). Specifically: when plan adherence checker finds recurring patterns, or when review identifies issues traceable to prompt instructions.

**Flow:**

```
Claude reviews → identifies template-caused issue
    → constructs consultation prompt with:
        - The specific failures (from review-result.md)
        - The relevant template excerpt
        - The module's output excerpt
        - Question: "What in this template caused this?"
    → sends via ask-gemini bridge
    → Gemini responds with:
        - root_cause: why the template produced this
        - proposed_change: specific FIND/REPLACE for template
        - scope: this_module | all_modules
        - action: rebuild | fix
    → result saved to orchestration/{slug}/consultation-{N}.md
    → if scope=all_modules: saved for human approval before applying
    → if scope=this_module + action=fix: applied, re-validated
```

**Template:** `claude_extensions/phases/gemini/consultation.md`

**CLI:** `--consult` flag on build_module_v5.py runs consultation on the latest failure instead of rebuilding.
