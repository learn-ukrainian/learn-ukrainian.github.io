The three big wins (~85KB / 44% of total)

Win #1 — KNOWLEDGE_PACKET is rendered TWICE (~27KB pure waste)

The template substitutes {KNOWLEDGE_PACKET} at line 456 ("## Knowledge Packet") AND {KNOWLEDGE_PACKET} at line 791 ("## Full Wiki Context (source of truth for citations)"). Same content, two locations. In the my-morning render:

  • Line 1201: ## Knowledge Packet → 13.3KB of wiki content
  • Line 1769: ## Full Wiki Context (source of truth for citations) → SAME 13.3KB

Action: remove the second substitution. Keep "## Knowledge Packet" only. Update the prose at line 791 to reference back ("see §Knowledge Packet above for citations").

Risk: zero. Pure copy-paste artifact from template evolution.

Saves: ~13.3KB on this module; up to ~27KB on larger modules where the manifest is fuller.

Win #2 — LESSON_CONTRACT §3 Component inventory overlaps with COMPONENT_PROPS_SCHEMA + Activity Authoring Fields section (~14KB redundancy)

Three sections specify activity component structure:

  • ## 3. Component inventory — 1:1 mapping (11.3KB) — from LESSON_CONTRACT, describes React component → MDX mapping
  • ## Activity Authoring Fields (mandatory) (2.9KB inline + COMPONENT_PROPS_SCHEMA substitution)
  • ## Activity Types and the INLINE / WORKBOOK split (mandatory) (1.0KB)

The writer needs the AUTHORING fields (the YAML/JSON shape consumed by scripts/yaml_activities.py) — NOT the React component prop names. LESSON_CONTRACT §3 was explicitly noted in _render_component_props_schema docstring as the source of past Gemini failures (writer used React names because it saw both). The writer SHOULD see only the authoring-field whitelist.

Action: drop LESSON_CONTRACT §3 from the writer prompt (move to docs/best-practices/v7-design-and-corpus.md as appendix; writer fetches on demand via @-path if needed for novel activity types). Keep COMPONENT_PROPS_SCHEMA + Activity Authoring Fields.

Risk: low. The authoring fields are what the parser checks; React-component knowledge is downstream MDX-render concern that the writer doesn't act on.

Saves: ~11.3KB.

Win #3 — NORTH_STAR philosophy + LESSON_CONTRACT framing (~22KB)

The writer prompt opens with the full NORTH_STAR (16.6KB across WHO/WHY/HOW/VOICE/SHIPPABLE/WRONG sections) and LESSON_CONTRACT (28.1KB across 7 numbered sections + resolved-policy block). Most of this is project-philosophy and architectural framing the writer doesn't act on at write-time — it's already been baked into the plan, the wiki obligations manifest, the implementation map, the immersion rule.

What the writer actually needs from these docs at write-time:

  • The 4 artifact contract (what files to produce) — keep, ~1 page
  • The 4-tab MDX shape — keep, ~1 paragraph
  • Tone / voice constraints (no English meta-narration) — already restated in §Tone and immersion (mandatory) inline. KEEP that, DROP NORTH_STAR § VOICE.
  • "What we will not ship" — already enforced by deterministic gates. The writer doesn't need to re-read the philosophy; the gates fail it if it tries.

Action: compress NORTH_STAR + LESSON_CONTRACT down to a single "## Writer charter (1 page)" section: the 4 artifact contract + the 4-tab shape + a 5-line voice anchor + a reference to docs/north-star.md + docs/lesson-contract.md for the full versions. Target: NORTH_STAR + LESSON_CONTRACT combined down from 50KB → 5KB. (~45KB saved, but realistic post-compression is ~40KB after we keep the load-bearing constraints inline.)

Risk: medium. The philosophy text isn't waste, but the writer LLM doesn't ground its output in philosophy — it grounds in the plan + manifest + immersion rule + gate constraints. If we cut and the writer regresses on voice, PR-C2 adds back the specific voice anchors that fired.

Saves: ~40KB (conservative; could be ~45KB).

Risks per cut

Risk: NORTH_STAR / LESSON_CONTRACT compression breaks voice or shape

Mitigation: Validation build IS the falsifier. If a1/my-morning regresses on voice (e.g. meta-narration creeps back, immersion drops), PR-C2 restores the specific anchors that fired. The compressed Writer Charter section will include the 5 anchors most likely to fire (no English meta-narration, no warm-up, immersion-ratio binding, voice as content not container, no "let us X").

Risk: dropping pre-emit bad_form_audit + activity_split_audit causes writer to skip self-check, ship more failures

Honest reality: the deterministic gates (vesum_verified, russianisms_clean, activity_split per #2238) FAIL the build on the same failures the audit was supposed to prevent. The audit was insurance against the gate; redundant. The cost of dropping it is one extra correction round on the build (not a worse module). Net: build is slower in failure cases, no quality regression in success cases.

Risk: LESSON_CONTRACT §3 component-inventory move to appendix means writer doesn't see component shapes

Mitigation: COMPONENT_PROPS_SCHEMA substitution already covers the authoring fields the writer acts on. The React-component prop names in §3 are downstream MDX-render concerns the writer never touches. The original docstring on _render_component_props_schema literally says Gemini failed because it saw React props. Removing them is the correction, not a regression risk.

Risk: the 40KB target is unreachable per the math above

Mitigation: reframe the target as raw-template 40KB (66KB → 40KB on scripts/build/phases/linear-write.md) + rendered ~80KB. The 40KB-rendered ambition would require shrinking the per-module data (KNOWLEDGE_PACKET / WIKI_MANIFEST), which is a separate engineering project (PR-D candidate: knowledge-packet diet). Don't try to hit 40KB rendered in PR-C.

Per-rule firing telemetry (added scope inside PR-C)

Per the strategic discussion: each rule statement gets a stable tag (e.g. #R-VESUM-ALL-WORDS, #R-BAD-FORM-MARKER, #R-NO-META-NARRATION) and the pipeline emits a writer_rule_fired JSONL event when a gate detects the rule was violated. After 20-30 builds we'll have empirical data showing which rules actually fire vs which are dead weight. PR-C2 (if needed) or future strip cycles use this data instead of gut.
