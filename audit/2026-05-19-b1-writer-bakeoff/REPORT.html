1.Headline findings

Three hypotheses tested. Two disconfirmed at one level then refined or confirmed at another. One adapter-layer bug found and fixed.

1. DeepSeek-v4-pro hypothesis (carried from A1 bakeoff): DISCONFIRMED at medium · CONFIRMED at xhigh

At B1's 4000-word target on b1-052 genitive-nuances, deepseek-tools at effort=medium emits all 7 plan_reasoning + 2 audit lines then stops before any artifact fence — 9 KB scaffolding-only, opposite of A1 where deepseek shipped the best content. Bumping to effort=xhigh recovers full artifact emission: 7 plan_reasoning + 4 fences + end_gate, 38,940 chars (largest of any writer tested). The effort-budget axis is real at B1+ where it was invisible at A1.

2. Codex-tools effort hypothesis: DISCONFIRMED

Codex CLI exits with returncode=1 in under 1 second with empty stderr, before creating a session/rollout file. Both effort=high (default) and effort=xhigh produce the identical sub-second crash. This is a runner-layer interaction bug, not a model-quality or reasoning-budget issue. Filed as #2159. Manual cat prompt.md | codex exec <same-flags> - succeeds with the exact runner flags, so the failure surface is the agent_runtime's PTY-stdout + pipe-stdin path on 210K-char prompts specifically.

3. Gemini-tools namespace bug: FOUND + FIXED

Gemini CLI 0.42.0 emits MCP tool calls with single-underscore mcp_sources_search_text instead of the canonical double-underscore mcp__sources__search_text used by claude / codex / hermes-backed writers. The pipeline's writer-trace-isolation gate hardcoded the double-underscore form, so all 14 of gemini's real MCP calls were misclassified as wrong_tool_family. Fixed in commit 0fc0f0d427: both prefixes now accepted; gemini's actual wrong-tool-family calls (built-in run_shell_command, update_topic) are still correctly flagged.

Production B1+ writer remains claude-tools (Opus 4.7 xhigh) — the only writer making real writer-time MCP linguistic-verification calls (24 in this run vs 0 for deepseek-xhigh / qwen). DeepSeek-tools at xhigh is the viable metered fallback for the post-2026-06-15 era when the Claude dispatch lane sunsets, but reviewer-phase MCP must compensate for the writer-time verification gap.

5.Gemini namespace fix shipped

Predecessor finding: gemini-tools fails two gates on the bakeoff. Investigation root-caused it to a gemini-cli 0.42.0 convention change. Fix shipped in commit 0fc0f0d427.

Root cause

Gemini CLI 0.42.0 changed the emitted MCP tool name convention from canonical double-underscore mcp__sources__search_text (per the MCP spec used by claude, codex, and the hermes-backed writers) to single-underscore mcp_sources_search_text. The pipeline had two surfaces that hardcoded the canonical form:

SurfaceFile:linePre-fix behavior on gemini
_normalize_tool_name()linear_pipeline.py:2102Split on __; single-underscore names fell through, raw name didn't match WRITER_TOOL_NAMES → call silently dropped from tool_calls_total → MCP_TOOLS_NEVER_INVOKED fired
classify_writer_trace()linear_pipeline.py:1989Single startswith(WRITER_ALLOWED_TOOL_PREFIX) against "mcp__sources__"; gemini's mcp_sources_* calls didn't match → wrong_tool_family fired

Fix

Introduced WRITER_ALLOWED_TOOL_PREFIXES as a tuple of accepted MCP-sources prefixes:

WRITER_ALLOWED_TOOL_PREFIXES = (
    "mcp__sources__",  # canonical (claude, codex, deepseek, qwen, grok)
    "mcp_sources_",    # gemini-cli 0.42.0+ single-underscore convention
)

_normalize_tool_name now also strips the mcp_sources_ prefix when present without __. The classify check uses any(tool_name.startswith(p) for p in WRITER_ALLOWED_TOOL_PREFIXES).

The fix doesn't unblock gemini as a writer; it stops misclassifying its legitimate MCP traffic. Disabling gemini-cli's built-in tools is a separate problem (the CLI offers --allowed-mcp-server-names but no equivalent built-in disable flag).

7.Writer-MCP-invocation gap

Three writers shipped full artifacts. Only one made real MCP linguistic-verification calls at writer-time. The other two wrote intent-traces and skipped the calls.

WriterOutputverification_trace blocksReal MCP callsGap?
claude-tools (Opus 4.7)54 KByes24none
deepseek-tools xhigh38 KByes (detailed)0scaffolding only
qwen-tools medium31 KByes0scaffolding only

The intended behavior is that verification_trace declares the calls the writer plans to make AND will then execute. Deepseek and qwen treat these blocks as documentation and skip the actual tool invocation. Hypotheses worth investigating:

  1. Adapter-side: hermes-backed adapters (deepseek-tools, qwen-tools, grok-tools) wire MCP servers through ~/.hermes/config.yaml rather than per-call tool config. If hermes' MCP exposure has visibility issues for these specific models, the writer can't invoke even though the prompt says to.
  2. Prompt-side: the prompt may inadvertently treat verification_trace as "describe what you would do" rather than "do it then describe what you did". Claude's training apparently distinguishes; smaller open models may not.
  3. Tool-discovery: deepseek/qwen may not even see the MCP tools in their context — hermes-side server start-up may fail silently for some models.

This matters for routing decisions: if writer-time MCP is the cost of using claude-tools, and the post-2026-06-15 fallbacks can't replicate it, the reviewer phase needs hardening to catch what the writer would otherwise verify pre-emission.
