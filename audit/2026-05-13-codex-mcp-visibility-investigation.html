#1900 — Codex MCP catalog visibility investigation

Investigator: Claude Opus 4.7 (headless dispatch, read-only) · Mode: diagnosis-only (NO code changes) · Dispatch SHA at run: 30dfc4ab6b · Bakeoff SHA under test: 5a72a31f9b · Date: 2026-05-13
TL;DR.
  1. Codex-tools did call MCP tools — 8 successful calls (verify_words×5 verifying 121 Ukrainian forms, all FOUND in VESUM; search_text×3 retrieving the exact textbook pages the build cited). It out-tool-called claude-tools by 2× in this bakeoff.
  2. The pipeline reported tool_calls_total=0 because of a rollout-binding bug in scripts/agent_runtime/adapters/codex.py::_rollout_matches_plan: the matcher returns False on the FIRST response_item/message role=user whose text does not equal plan.stdin_payload. Codex CLI 0.130.0 injects AGENTS.md as such a message before the real user prompt, so the matcher always trips on AGENTS.md and never reaches the actual prompt.
  3. The fix is mechanical (~10–20 LOC): change the matcher from fail-fast-on-first-mismatch to iterate-all-user-messages-and-return-true-on-any-match. Symptom is fully consistent with the 3-bakeoff tool_calls_total=0 pattern; nothing about the writer prompt, Codex catalog, or MCP config is broken.

Decision-grade question — answered

"Is tool_calls_total=0 because (a) Codex never received the MCP tool definitions, (b) Codex received them but ignored them, OR (c) Codex called the tools but the telemetry collector missed them?"

Answer: (c) — telemetry collector missed them. Specifically, the codex adapter's rollout-binding logic fails to associate the writer's rollout JSONL with the writer's InvocationPlan, so parse_response returns tool_calls=[] even though the rollout on disk contains 8 successful MCP function calls.

Verdict — RULED IN. Codex's CLI 0.130.0 injects AGENTS.md as a response_item/message role=user before the user's actual prompt; the pipeline's _rollout_matches_plan matcher returns False on this first synthetic user-role message; _select_rollout_for_plan therefore returns None; _read_latest_rollout_trace returns empty; parse_response returns tool_calls=[]; the pipeline fires MCP_TOOLS_NEVER_INVOKED despite Codex having called 8 MCP tools successfully.

Why the existing test missed this

In tests/test_agent_runtime.py:1377–1396 the rollout fixture contains ONLY event_msg/user_message + event_msg/task_complete. No response_item/message role=user AGENTS.md envelope. Production Codex CLI 0.130.0 ALWAYS emits the AGENTS.md envelope first (it is part of the codex-exec startup contract — if an AGENTS.md exists in cwd or its parents, Codex injects it). The test fixture exercises only the happy path.

Sibling-bug check: gemini.py adapter has its own session-file recovery but uses a different format (type:"gemini", toolCalls:[…] per scripts/agent_runtime/tool_calls.py:46–49), not Codex's nested-event shape. No equivalent rollout-matching matcher to misbehave. Bug is Codex-specific.

Root cause

scripts/agent_runtime/adapters/codex.py::_rollout_matches_plan uses fail-fast return-on-first-match logic. Codex CLI 0.130.0 emits an AGENTS.md-bearing response_item/message role=user event before the user's actual prompt. The matcher trips on AGENTS.md, sees the text differs from plan.stdin_payload, and returns False before ever reaching the real prompt event — so _select_rollout_for_plan reports no match, parse_response falls back to tool_calls=[], and the pipeline's MCP_TOOLS_NEVER_INVOKED guard fires despite Codex having successfully invoked 8 MCP tools.

This explains all 3 bakeoff occurrences (2026-05-06, 2026-05-08, 2026-05-12) — none of the prompt iterations could ever clear the gate, because the failure is in the runtime adapter's telemetry collection, not the prompt or the writer behavior.

Proposed fix design

Change _rollout_matches_plan to iterate ALL candidate user-role messages and return True if ANY of them match plan.stdin_payload (rstripped); only return False if the whole file has been scanned without a match. Optional refinement: prefer the canonical event_msg/user_message branch over response_item/message role=user (only the former is guaranteed to be a real user prompt; the latter is a stream-replay format Codex uses for both injected context AND the real prompt).

Required test additions:

  • Augment the existing fixture in tests/test_agent_runtime.py with a response_item/message role=user AGENTS.md envelope before the event_msg/user_message. Test should still match.
  • Add a regression case where a rollout has TWO response_item/message role=user events (AGENTS.md + real prompt) and stdin_payload matches the second. Test should match.
  • Add a fingerprint test using a real rollout snippet from a recent Codex 0.130.0 dispatch as a checked-in fixture so future Codex CLI envelope changes break the test loudly.

Scope: ~10–20 LOC in the matcher + ~30 LOC of test fixtures. Mechanical fix — suitable for a Codex dispatch with full reproducer in the brief. No architectural decision required.

What I did NOT verify

  • Are reviewer/dispatch calls affected? The same matcher is used by every Codex invocation through scripts/agent_runtime/runner.py. So any non-writer Codex call (codex reviewer in pipeline, delegate.py dispatch --agent codex, ab discuss codex rounds) that depends on extracting tool_calls from the rollout is silently dropping those telemetry events. I did not enumerate every dispatch site and verify the impact on each — but the code path is shared, so the bug is broad. This is a key follow-up question for the orchestrator.
  • Why the early bakeoffs (2026-05-06, 2026-05-08) also showed tool_calls_total=0 — was the matcher already buggy on those SHAs, or was the original buggy version a different codepath? Git blame on the matcher block (codex.py:670–717) points to a5f96a2d28 "fix: harden v6 style-heal and verify pipeline" dated 2026-04-15. That is before all three bakeoffs, so the bug has been in production since mid-April. I did not separately re-check 2026-05-06 / 2026-05-08 rollouts to confirm Codex was already calling tools on those bakeoffs — strongly likely given identical adapter + identical Codex CLI version (0.130.0) — but the evidence base is the night bakeoff.
  • Whether the AGENTS.md injection is conditional on the cwd's filesystem layout (i.e. does Codex skip the injection when no AGENTS.md is found upward from cwd?). The bakeoff cwd was a worktree where AGENTS.md likely was present in the parent tree, so this rollout always has the envelope. I did not test the no-AGENTS.md case — if Codex skips the envelope there, the bug might not manifest in those calls. But the writer cwd in normal builds (curriculum/) does have AGENTS.md upward, so the writer pipeline always trips.
  • Whether the same rollout binding works for the bakeoff's claude-tools writer — claude doesn't use rollout binding (its tool_calls come from stream-json stdout), so the bug is exclusively codex-side. I did not look for an analogous bug in gemini.py session-file recovery, but the data shape there is different (type:gemini, toolCalls:[…]), so unlikely to share the same root cause.
