What the Multi-UI ADR proposes canonical design, summarised

QTopicCodex-Desktop-relevant proposal
Q1What is a participant?Four columns: agent_family=codex, ui_surface=desktop, persistent client_id, per-session instance_id. Display alias codex:desktop:8f91c2 is non-canonical. Persisted at ~/.config/learn-ukrainian/agent-bridge/instance.json.
Q2How does Desktop discover a discussion?SSE replay: GET /api/comms/channels/{channel}/events?since={event_id}. MCP resource registration (NOT long-running tool call) for Desktop's model-turn semantics. Pull/inbox fallback remains.
Q3Who can claim a turn?Claim-gated round posts. Default participant_scope=family — Codex CLI and Codex Desktop both satisfy a "Codex" slot, first to claim wins. Restart recovery via persisted client_id; window-vs-window collision via per-session instance_id; takeover via /claims/{id}/seize after missed keepalives.
Q6Lease semanticsDesktop holders get a 5-minute lease with 30 s keepalives (vs 90 s for CLI subprocesses) — accommodates human-in-loop visual inspection. Visible expires_at.
Q7IdempotencyEvery post + claim takes idempotency_key keyed by (client_id, key). Survives Desktop reconnect. Required for multimodal uploads (5-min window).
Q12Multimodal attachmentsFirst-class blobs + message_attachments tables. Filesystem at .ab/channels/blobs/{sha256}.{ext}. Atomic POST /messages-with-attachments. attachment://{id} markdown URI scheme. Hard-required for round replies that need image/audio.

What this MVP does NOT do: no 4-tuple identity (still a flat from_agent string), no claim-based round participation, no SSE replay, no atomic multimodal post, no schema migration. All that's deferred to the full Multi-UI ADR implementation.

Why the MVP is enough for now: the goal here is orchestrator → Codex Desktop async messaging (briefs in, status out). The MVP delivers that. The full ADR adds turn-taking and multimodal — needed for ab discuss participation and graphics-rich module work, but neither is on the critical path right now.
