# Starlight Frontend — learn-ukrainian

The curriculum site is built with [Astro Starlight](https://starlight.astro.build)
and deployed to GitHub Pages from `main`. All module MDX lives in
`src/content/docs/` and is generated by the Python build pipeline — NEVER
hand-edit MDX files.

## Project layout

```
starlight/
├── public/                          # Static assets (favicons, fonts, OG images)
├── src/
│   ├── assets/                      # Images used in MDX
│   ├── components/                  # ~50 React components (interactive exercises)
│   │   ├── activities/              # Quiz, Fill-in, Match, etc. — rendered from YAML
│   │   ├── flashcards/              # Flashcard widget for словник tab
│   │   └── navigation/              # Sidebar, breadcrumbs, tab bar
│   ├── content/
│   │   ├── docs/                    # Generated MDX (DO NOT HAND-EDIT)
│   │   │   ├── a1/ a2/ b1/ b2/ c1/ c2/
│   │   │   └── seminars/            # HIST, BIO, LIT, ...
│   │   └── config.ts                # Content collection schema
│   ├── styles/                      # Global CSS (tab layout, typography)
│   └── content.config.ts
├── astro.config.mjs                 # Starlight + integrations
├── package.json
└── tsconfig.json
```

## How module MDX is structured

Every module MDX has **4 tabs** defined by a shared layout component:

1. **Урок (Lesson)** — prose, examples, grammar explanations
2. **Словник (Vocabulary)** — flashcard deck (vocabulary YAML)
3. **Зошит (Workbook)** — interactive activities (activities YAML)
4. **Ресурси (Resources)** — textbook links, videos, references

The Python publish step (`scripts/build/v6_build.py` → `step_publish`) reads
the content `.md`, vocab YAML, activities YAML, and plan references, and
emits a single MDX file that imports the activity/flashcard components and
embeds them in the right tabs.

## Activity components

Activities are JSX renderers fed by YAML. Supported types live in
`src/components/activities/*.tsx` and their schemas are registered in
`scripts/config_tables.py` (server side) plus matched by the publish step
when choosing which component to render.

Full matrix: `docs/best-practices/vocabulary-activity-standards.md` and
`docs/ACTIVITY-YAML-REFERENCE.md`.

## Commands

Run from `starlight/`:

| Command                   | Action                                           |
| :------------------------ | :----------------------------------------------- |
| `npm install`             | Install dependencies                             |
| `npm run dev`             | Local dev server on `http://localhost:4321`      |
| `npm run build`           | Build production site to `./dist/`               |
| `npm run preview`         | Preview built site locally                       |
| `npm run astro check`     | Type-check MDX + content collections             |

The `services.sh start` helper in the repo root launches `npm run dev` in
parallel with the RAG MCP server and monitor API, so you can run the whole
stack with one command.

## Do not hand-edit

- `src/content/docs/**/*.mdx` — generated by the publish pipeline. Edits
  will be overwritten on the next build.
- `data/` — generated SQLite + JSONL caches.

Fix the problem upstream: in the plan YAML, the prompt template, or the
publish-step code — never in the emitted MDX.
