Gemini-tools deep review

Root-cause audit for the 2026-05-08 Claude/Gemini diagnostic bakeoff.
Date: 2026-05-09
Thread: gemini-tools-deep-review-2026-05-09
Status: root cause found
Scope: report only, no code changes
This is the Gemini-side counterpart to audit/codex-tools-review-2026-05-08/REPORT.md. Codex-tools failed because Codex CLI rejected a Claude-format MCP config field. Gemini-tools failed for a different reason: the writer subprocess ran from an artifact directory where Gemini CLI had no MCP server configured.

Executive Summary

Root cause: --allowed-mcp-server-names sources is only an allow-list. It does not define or load the sources server. Because v7 runs Gemini from audit/.../gemini/, where no Gemini MCP settings file exists, Gemini starts with no MCP servers configured and therefore cannot call mcp__sources__verify_words or mcp__sources__search_text.

E9. Telemetry parser misses Gemini JSONL tool calls

Even the built-in list_directory call did not reach writer_tool_calls.json. The adapter reason is visible in code:

# scripts/agent_runtime/adapters/gemini.py

    return json.dumps(data, ensure_ascii=False)
except Exception:

Current Gemini sessions are JSONL: the matched file has 8 JSON objects on separate lines. Loading the whole file as one JSON document throws and returns an empty trace. This is a secondary observability bug, not the MCP root cause.

Root Cause Statement

Gemini-tools produced writer_tool_calls.json: [] because the Gemini writer subprocess was launched from module_dir, an artifact directory with no .gemini/settings.json. Gemini CLI therefore had no configured MCP servers for that workspace. The pipeline's pre-flight checked only that sources exists in repo .mcp.json, then passed --allowed-mcp-server-names sources, which filters a catalog but does not create one. The model-side catalog contained built-in tools such as list_directory, but not mcp__sources__verify_words or mcp__sources__search_text.

Recommended Fix

Option A - run Gemini writers from repo root and keep artifact paths explicit. For Gemini only, call invoke_writer(..., cwd=PROJECT_ROOT, tool_trace_path=module_dir / "writer_tool_calls.json"). The writer output already returns via stdout and is written by Python, so the artifact directory does not need to be the subprocess cwd. This is likely the smallest fix, but verify that no writer prompt relies on relative output paths.

Option B - stage Gemini MCP config into the module directory before invocation. Copy or synthesize module_dir/.gemini/settings.json from repo root before running Gemini. This preserves the current cwd shape but introduces generated config files in artifact dirs, which must be cleaned or ignored.

Option C - add adapter-level config path support if Gemini CLI exposes it. I did not find a --mcp-config equivalent in gemini --help or gemini mcp --help. If such a hidden/env mechanism exists, use it; otherwise do not build on this assumption.

Recommended: Option A, plus a positive runtime gate already added by PR #1813 should remain in place for all *-tools writers. The gate is what turns future "configured but no model-side calls" failures into hard failures instead of theatrical output.

Patch Sketch

If Option A is accepted, the code patch is small and belongs in a separate fix dispatch:

# scripts/build/v7_build.py
writer_cwd = linear_pipeline.PROJECT_ROOT if writer == "gemini-tools" else module_dir
writer_output = linear_pipeline.invoke_writer(
    prompt,
    writer,
    cwd=writer_cwd,
    tool_trace_path=module_dir / "writer_tool_calls.json",
    stdout_silence_timeout=args.writer_timeout,
)

That sketch is intentionally not committed here; this dispatch is report-only.

Test Plan

  • Unit test: for gemini-tools, assert v7 passes repo root as the invocation cwd while preserving tool_trace_path=module_dir / "writer_tool_calls.json".
  • Adapter test: create a JSONL Gemini session with a toolCalls entry and assert GeminiAdapter.parse_response(...).tool_calls captures it. This covers the secondary telemetry bug.
  • Smoke check, no full bakeoff: from repo root, gemini mcp list must report sources connected; from a module artifact dir it currently reports no MCP servers, proving the fix must avoid that cwd or stage config there.
  • User-run bakeoff verification after fix: gemini/writer_tool_calls.json should contain at least one mcp__sources__verify_words or mcp__sources__search_text call, and the runtime gate should fail if it does not.

What This Is Not

  • Not the Codex type="streamable-http" bug. Gemini's LKG shape is type="http" at repo root.
  • Not a missing wiki packet. The packet is byte-identical across writers.
  • Not primarily sandbox denial. The model had built-in filesystem tools and used list_directory.
  • Not solved by --allowed-mcp-server-names. That flag cannot load a server that the workspace did not configure.

Open Questions

  • Does Gemini CLI have an undocumented environment variable or policy setting that points to a non-workspace MCP config? I found no public flag in local help output.
  • Should v7 standardize all writers on repo-root cwd and make artifact paths explicit, or only patch Gemini? I recommend the narrow Gemini-only fix first to avoid disturbing Claude/Codex behavior.
  • Should writer_prompt.md be written before invocation so failed runs still preserve the prompt artifact? The Gemini run lacked it because v7 writes the prompt only after invoke_writer returns.
